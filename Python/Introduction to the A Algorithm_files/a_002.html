                    amitp Mod Leo 8 years ago

                    Yes, I considered implementing on the server side, but there are two big disadvantages:

                    (1). It means I need a more complicated server, and have to deal with load spikes such as when the page is posted on reddit or hackernews and I get 50k hits all at once. Right now I have one "micro" amazon instance that serves static html, and it can handle all the traffic. With static html I also have the flexibility of using s3 or some other system. Static html can also be archived and cached; it's less fragile.

                    (2). There's some latency for calculating results and getting them back. Unless I have servers all over the world, some users may have a latency of 300ms or more. That means as you drag the slider, I could not make a smooth animation of the recalculation. Right now with client side calculations it can run in under 10ms, which allows for smooth animations.

                    It might seem like running the algorithms on the server would be useful if the algorithms were complicated, but because of problem (1), I would not be able to run the algorithms if there is a load spike! The total computing power in the browsers is larger than the total computing power on my servers. Running the algorithms on the server might be useful if the data is big, but so far these tutorials have small data. Maybe there will be a tutorial in the future where I need larger data.

                                                  amitp Mod bobo 8 years ago

                                                  bobo, that's a really good question. It's something I didn't explain adequately. Both A* and Dijkstra's Algorithm can reduce the priority of a node as they explore and find better paths to the node. As long as a node is in the frontier, it can be updated with better priorities. In this case, A* hasn't found the best path to [3,1] (because it's efficientit's not spending as much time exploring paths to things that don't matter), and its cost_so_far is 16. It thinks the best path to [3,1] is by going west from [4,1]. If A* had time to explore more, it'd find a better path to [3,1] is by going north from [3,2], with a cost_so_far of 14. You can trigger this by blocking the shortest path and you'll see the costs drop.

                                                                                                                amitp Mod Ethan Allen 2 years ago

                                                                                                                I think that's where the usual A* tutorials break down A* is only one piece of the puzzle and there can be *lots* of different approaches layered on top. A pure A* idea might be to generalize from location nodes to state nodes. Then you have a graph of states, where the state would include x, y, and remaining movement for each unit. This leads to a lot more nodes, but it'd give you more precise paths. Another idea would be to have A* explore a graph of movements you can make in one turn, so A* would see one node per turn. Then you'd construct the nodes (in the neighbors function) to be all possible movements you can make this turn, taking into account movement cost and maxmovement. You could do this step with BFS or Dijkstra's. But this too might lead to a large number of nodes. I expect that in practice most games give up on optimal paths and instead settle for reasonable paths using some game-specific logic to decide how to move combined with A* for the overall pathfinding.

                                                                                                                I agree, it's seldom talked about, but I think that's because it's not a general universal solution but instead a game-specific solution. I'm guessing it's something devlogs might have but tutorials don't.

                                                                                                                                                    amitp Mod Pheo 2 months ago

                                                                                                                                                    Thanks! Yes, you can do a breadth-first search with a custom "early exit". For normal pathfinding we do early exit when we find the destination node. But what you want instead is to stop searching when you reach some "range".

                                                                                                                                                    There are two things we could track in BFS. One is the "parent" pointer, used for reconstructing paths. The other is the distance, which is what you want here. If you want all cells up to distance 4 then when you are adding neighbors to the queue, do not add any neighbors that have distance greater than 4. That way the queue will contain *only* the nodes distance 4 or less.

                                                                                                                                                    Then you will not have to run A* to check distances, and it will run very very fast.

                                                                                                                                                    I have a demo of this here: https://www.redblobgames.co... . It is essentially the same as the BFS algorithm but written slightly differently.
